# Builder Agent (The Engineer)

## Role Overview
The **Builder Agent** executes the isolated technical tasks defined by the Planning Agent. Its function is to write functional application code, create localized unit tests, and write inline documentation in adherence to the established blueprint.

## System Trigger
Triggered by a webhook when a new task is detected in a merged `task.md` blueprint, or when an engineering ticket is assigned to it in the Git repository.

## Inputs (from Single Source of Truth)
- **Technical Blueprint**: The architectural diagram, API contracts, and task breakdown generated by the Planning Agent.
- **Codebase Context**: Relevant files from the main branch required to execute the assigned task.
- **Failure Feedback Loops**: If the QA or Security (Red Team) Agent rejects a previous commit, the Builder Agent takes the CI failure logs as direct input.
- **Historical Fixes**: Queries the **Learning Agent** for common bug fixes, high-value regression patterns, and module-specific lessons established in this area of the codebase.

## Outputs (to Single Source of Truth)
- **Application Code**: Source files implementing new features or fixing bugs.
- **Unit Tests**: Executable test suites scoped to the new code.
- **Git Action**: Opens an "Implementation Pull Request" pointing to the main branch. Subsequent failures from QA or Security result in new commits to this exact same PR.

## Interaction with Other Agents
- **QA & Security Agents**: Works iteratively to resolve any issues raised by these auditing agents before the PR is marked ready for human review.
- **Platform Agent**: May request IaC changes if the application code requires a new infrastructure resource (e.g., a new S3 bucket or database table).
- **Governance Layer**: Receives policy outcomes and evidence summaries indirectly through the PR workflow, but cannot bypass governance by "fixing forward" without passing the required gates.
