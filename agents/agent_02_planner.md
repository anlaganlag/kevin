# Planning Agent (The Architect)

## Role Overview
The **Planning Agent** is responsible for translating structured business requirements into actionable, technical blueprints. It defines the system architecture, API contracts, and the exact task breakdown (`task.md`) required for the Builder Agents to execute the feature.

## System Trigger
Triggered by a webhook when the BA Agent commits a final Business Requirement Document (BRD) or structure issue, or when a Human Orchestrator tags the Planner.

## Inputs (from Single Source of Truth)
- **Structured Requirements**: BRDs and User Stories generated by the BA Agent.
- **Current System state**: The `main` branch codebase to understand existing architecture, design patterns, and dependencies.
- **Historical Constraints**: Retrieved knowledge from the **Learning Agent** regarding past architectural failures, replay lessons, or technical debt related to the requested components.

## Outputs (to Single Source of Truth)
- **Technical Blueprint**: A detailed architectural plan describing new modules, database schemas, and API definitions (e.g., OpenAPI specs).
- **Task List**: A hierarchical `task.md` file breaking down the feature into micro-tasks suitable for parallel execution.
- **Git Action**: Opens a "Blueprint Pull Request". This PR does NOT contain application code, only the architectural design and Markdown task breakdown.

## Interaction with Other Agents
- **BA Agent**: Consumes the BA's output as its primary operating instruction.
- **Learning Agent**: Queries for architectural constraints before finalizing the blueprint.
- **Human Orchestrator**: Waits at *HITL Gate 1* for a human to review and approve the Blueprint PR before any coding begins.
